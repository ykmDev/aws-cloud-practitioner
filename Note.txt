    1) Cloud Computing
      def: the practice of using a network of remote servers hosted on internet
        to store, manage, process data rather than local/personal server, computer.
          On-Premise:
            You own the servers, hire IT people, pay or rent the real-estate, take all risk.
          Clud Providers:
            someone else own servers, hire IT people, pay or rent real-estate,
            you are responsible for your configuration cloud services and code.
      evaluation:
        dedicated server:
          one physical machine dedicated to single a business, run single web-app.
          Note: very expensive, high maintenane, high Security.
        virtual private server(VPS):
          one physical machine dedicated to a single business, 
          one physical machine is virtualized into a sub-machines
          runs Multiple web-apps.
          Note: better utilization and isolation of resources.
        shared hosting:
          one physical machine shared by hundred of businesses,
          Note: very cheap, limited functionality, poor isonlation.
        Cloud Hosting:
          Multiple physical machines that act as one system.
          abstracted into multiple cloud services.
          Note: flexible, scalable, cost-effective, high configurablility.

        Evolution of Computing:
          dedicated:
            a physical server wholly by a signle customer.
            can't Vertical scale
            replacing a server is very difficult
          vms:
            can run multiple virtual machines on one machine.
            Hypervisor is the software layer that you run the VMs
            a physical server shared by multiple customers
            multiple apps can result in conflicts in resource sharing.
          containers:
            virtual machine running multiple containers
            Docker daemon is the name of software layer.
            can maximize the utilize of the available capacity
            your containers share the same underlying OS so containers are more efficient than multiple vms.
            multiple apps can run side by side without being limited to same os requirements and will not cause conflicts during resources sharing.
          functons:
            are managed vms running managed containers.
            know as serverless compute.
            you upload a piece of code, choose th amt of memory and duration.
            only responsible for code and data.
            very-cost effective, only pay for time code running,
            cold starts is a side-effect of this setup.
        Type of Cloud Computing:
          SaaS:
            for customers
            a product that is run and managed by service provider,
            Don't worry about how the service is maintenaned.
            eg: salesforce, office 365, gmail.
          PaaS:
            for Developers
            focus on deployment, management of your apps.
            Don't worry about, provisioning, configuring or understanding the hardware or OS.
            eg: elastic beanstack, heroku, google app engine.
          IaaS:
            for admins
            the basic building for cloud IT. provides access to networking features, computers and storage.
            Don't worry about IT staff, data centers and hardware.
            eg: Azure, aws, oracle cloud.

          Six Advantages of Cloud:(aws original description of cloud benefits)
            1. Trade Capital Expense for Variable Expense
            အရင်လို server, storage, network equipment တွေကို upfront cost နဲ့ကြိုဝယ်စရာမလိုဘူး။ Cloud မှာတော့ Pay-as-you-go ဖြစ်လို့ သုံးသလောက်ပဲ ပိုက်ဆံပေးရတယ်။
            2. Benefit from Massive Economies of Scale
            Cloud provider တွေက customer အများကြီး share သုံးတာကြောင့် cost သက်သာတယ်။ ကိုယ်တိုင် server ဝယ်တာထက် unit cost ပိုနည်းတယ်။
            3. Stop Guessing Capacity
            Traffic ဘယ်လောက်လာမယ်ဆိုတာကြိုမှန်းဖို့မလိုဘူး။ လိုအပ်ရင် scale up လုပ်နိုင်တယ်၊ traffic ကျရင် scale down လုပ်နိုင်တယ်။
            4. Increase Speed and Agility
            Server, database, storage တွေကို minutes အတွင်း create လုပ်နိုင်တယ်။ Hardware order လုပ်ပြီး ရက်အတော်ကြာစောင့်စရာမလိုဘူး။
            5. Stop Spending Money on Running and Maintaining Data Centers
            Company က power, cooling, rack, hardware maintenance, security, networking စတာတွေကို ကိုယ်တိုင်မလုပ်တော့ဘူး။ အဲ့အစား customer နဲ့ business growth ကိုပို focus လုပ်နိုင်တယ်။
            6. Go Global in Minutes
            Application ကို region အများကြီးမှာ လွယ်လွယ် deploy လုပ်နိုင်တယ်။ ဥပမာ Amazon Web Services မှာ multiple regions ရှိလို့ Japan, Singapore, US, Europe စတဲ့နေရာတွေမှာ app ကို run လုပ်နိုင်တယ်။
        AWS Global Infrastructure:

            Performance ကောင်းတဲ့ application design လုပ်နိုင်တယ်

              Region

              Region ဆိုတာ geographical area တစ်ခုပါ။
              ဥပမာ Tokyo Region, Singapore Region

              Region တစ်ခုချင်းစီမှာ data center အများကြီးပါဝင်ပါတယ်။

              Availability Zone (AZ) ဆိုတာ Region တစ်ခုအတွင်းက independent data center group တစ်ခုဖြစ်ပါတယ်။

              ဥပမာ Region တစ်ခုမှာ AZ 3 ခုရှိနိုင်တယ်။ Application ကို multiple AZ မှာ run လုပ်ထားရင် data center တစ်ခု down သွားရင်တောင် service မပြတ်နိုင်ဘူး။

              Edge Location ဆိုတာ content delivery နဲ့ caching အတွက် user နီးနားမှာထားတဲ့ location တွေပါ။

              ဥပမာ Amazon CloudFront က Edge Location တွေကိုသုံးပြီး website, image, video တွေကို user နီးနားကနေ မြန်မြန်ပို့ပေးတယ်။

              Local Zone ဆိုတာ specific city တစ်ခုနီးနားမှာ low-latency service ပေးဖို့ region extension လုပ်ထားတာပါ။

              AWS မှာ compute, storage, database, AI, networking, analytics စတဲ့ service category အမျိုးမျိုးရှိပါတယ်။

              Security ပိုင်းမှာတော့ AWS က physical infrastructure security ကိုတာဝန်ယူပြီး customer က application, data, access control, configuration security ကိုတာဝန်ယူရပါတယ်။ ဒါကို Shared Responsibility Model လို့ခေါ်ပါတယ်။

                Traffic ဆိုတာ system ထဲကိုဝင်လာတဲ့ requests, users, data amount ကိုဆိုလို,
                Workload ဆိုတာ system က process လုပ်ရတဲ့ actual work ကိုဆိုလို

                  တစ်ဖက်မှာ user 10 ယောက်ပဲရှိပေမယ့် video rendering လုပ်နေရင် traffic နည်းပေမယ့် workload အရမ်းများနိုင်တယ်။

              ဥပမာ -
                Web server 1 လုံးကနေ 5 လုံးဖြစ်သွားတာ
                Traffic များလာရင် EC2 instance အသစ်တွေ auto create လုပ်တာ

              1 Server→5 Servers

              Horizontal scaling ကို scale out / scale in လို့လည်းခေါ်တယ်။
              Web applications တွေမှာ horizontal scaling ကိုပိုသုံးကြတယ်။
              AWS မှာတော့ horizontal scaling အတွက် Amazon EC2 Auto Scaling နဲ့ Elastic Load Balancing ကိုသုံးကြတယ်။

              Auto Scaling က traffic ပေါ်မူတည်ပြီး instance အသစ်တွေ create / remove လုပ်ပေးတယ်
              Load Balancer က user requests တွေကို server အများကြီးဆီ distribute လုပ်ပေးတယ်
            Note: Server တွေက request အရမ်းများလာရင် timeout, slow response, outage ဖြစ်နိုင်တယ်။ Auto Scaling က အဲ့ဒီ problem ကိုကာကွယ်ပေးတယ်
            Dynamic Scaling
              Dynamic scaling က real-time traffic change ကိုကြည့်ပြီး instance တိုး / လျှော့ လုပ်တာပါ။
            Predictive Scaling
              Predictive scaling က historical traffic pattern ကိုကြည့်ပြီး ကြိုတင် scale လုပ်တာပါ။
              ဥပမာ -
                Website က နေ့တိုင်း မနက် 9 နာရီမှာ traffic တက်တယ်
                Auto Scaling က 8:50 မှာတင် instance အသစ်တွေကြိုဖွင့်ထားမယ်
            Auto Scaling Group (ASG):
              ASG မှာ capacity limit 3 မျိုးသတ်မှတ်တယ် -
                Minimum = အနည်းဆုံး instance ဘယ်နှစ်လုံး run နေရမလဲ
                Desired = Normal traffic အတွက် ideal instance count
                Maximum = ဘယ်နှစ်လုံးထက်မပိုရဘူးဆိုတဲ့ limit
                ဥပမာ -
                  Minimum = 2
                  Desired = 2
                  Maximum = 10
                  2≤Instances≤10

                  ဒီ setup မှာ normal အချိန် 2 instances run မယ်။
                  Traffic တက်လာရင် Auto Scaling က 3, 4, 5 instance ဖြစ်အောင်တိုးမယ်။
                  အများဆုံး 10 instances ထိပဲတိုးနိုင်မယ်။
            Summary:
              Vertical Scaling = Server တစ်လုံးကို ပိုကြီးအောင်လုပ်တာ
              Horizontal Scaling = Server အရေအတွက်တိုးတာ
              Dynamic Scaling = Traffic တက်မှ auto တိုးတာ
              Predictive Scaling = Traffic တက်မယ့်အချိန်ကို ကြိုတွက်ပြီး တိုးတာ
        ELB:
          incoming traffic တွေကို EC2 instances အများကြီးကြားမှာ distribute လုပ်ပေးတဲ့ service
            Routing Methods:
              Round Robin
                Request တွေကို server တစ်လုံးပြီးတစ်လုံး အလှည့်ကျ distribute လုပ်တယ်။
                ဥပမာ -
                Request 1 → Server A
                Request 2 → Server B
                Request 3 → Server C
                Request 4 → Server A
              Least Connections
                Active connection အနည်းဆုံးရှိတဲ့ server ကို request ပို့တယ်။
                ဥပမာ server A မှာ 50 users ရှိပြီး server B မှာ 10 users ပဲရှိရင် request အသစ်ကို server B ဆီပို့မယ်။
              IP Hash
                Client IP ကိုကြည့်ပြီး user တစ်ယောက်ကို server တစ်လုံးတည်းဆီ consistent ပို့တယ်။
                ဥပမာ user တစ်ယောက်က session data တူညီနေဖို့ server တစ်လုံးတည်းကိုပဲရောက်နေစေချင်တဲ့အခါ သုံးတယ်။
              Least Response Time
                Response အမြန်ဆုံးပေးနိုင်တဲ့ server ကို request ပို့တယ်။
                ဥပမာ server A က 100ms နဲ့ပြန်တယ်၊ server B က 30ms နဲ့ပြန်တယ်ဆိုရင် request အသစ်ကို server B ဆီပို့မယ်။
