    1) Cloud Computing
      def: the practice of using a network of remote servers hosted on internet
        to store, manage, process data rather than local/personal server, computer.
          On-Premise:
            You own the servers, hire IT people, pay or rent the real-estate, take all risk.
          Clud Providers:
            someone else own servers, hire IT people, pay or rent real-estate,
            you are responsible for your configuration cloud services and code.
      evaluation:
        dedicated server:
          one physical machine dedicated to single a business, run single web-app.
          Note: very expensive, high maintenane, high Security.
        virtual private server(VPS):
          one physical machine dedicated to a single business, 
          one physical machine is virtualized into a sub-machines
          runs Multiple web-apps.
          Note: better utilization and isolation of resources.
        shared hosting:
          one physical machine shared by hundred of businesses,
          Note: very cheap, limited functionality, poor isonlation.
        Cloud Hosting:
          Multiple physical machines that act as one system.
          abstracted into multiple cloud services.
          Note: flexible, scalable, cost-effective, high configurablility.

        Evolution of Computing:
          dedicated:
            a physical server wholly by a signle customer.
            can't Vertical scale
            replacing a server is very difficult
          vms:
            can run multiple virtual machines on one machine.
            Hypervisor is the software layer that you run the VMs
            a physical server shared by multiple customers
            multiple apps can result in conflicts in resource sharing.
          containers:
            virtual machine running multiple containers
            Docker daemon is the name of software layer.
            can maximize the utilize of the available capacity
            your containers share the same underlying OS so containers are more efficient than multiple vms.
            multiple apps can run side by side without being limited to same os requirements and will not cause conflicts during resources sharing.
          functons:
            are managed vms running managed containers.
            know as serverless compute.
            you upload a piece of code, choose th amt of memory and duration.
            only responsible for code and data.
            very-cost effective, only pay for time code running,
            cold starts is a side-effect of this setup.
        Type of Cloud Computing:
          SaaS:
            for customers
            a product that is run and managed by service provider,
            Don't worry about how the service is maintenaned.
            eg: salesforce, office 365, gmail.
          PaaS:
            for Developers
            focus on deployment, management of your apps.
            Don't worry about, provisioning, configuring or understanding the hardware or OS.
            eg: elastic Beanstalk, heroku, google app engine.
          IaaS:
            for admins
            the basic building for cloud IT. provides access to networking features, computers and storage.
            Don't worry about IT staff, data centers and hardware.
            eg: Azure, aws, oracle cloud.

              High Availability (HA):
                High Availability ဆိုတာ service မပြတ်တောက်အောင် design လုပ်ထားတာပါ။
                အဓိက idea က single point of failure မရှိရဘူး။
                Single point of failure ဆိုတာ component တစ်ခုတည်းပျက်ရင် system တစ်ခုလုံး down သွားနိုင်တဲ့နေရာပါ။
                ဥပမာ -
                  Server တစ်လုံးတည်းရှိတယ်
                  Database တစ်လုံးတည်းရှိတယ်
                  Availability Zone တစ်ခုတည်းသုံးတယ်
                ဒီလိုဆိုရင် အဲ့ဒီ component ပျက်သွားတာနဲ့ application တစ်ခုလုံး down သွားနိုင်တယ်။

                Elastic Load Balancing ကိုသုံးရင် traffic ကို server အများကြီးကြား evenly distribute လုပ်နိုင်တယ်။

                ဥပမာ -
                Server A
                Server B
                Server C
                ရှိတယ်ဆိုရင် user requests တွေကို ELB က server ၃ လုံးဆီခွဲပို့ပေးမယ်။

                Server B ပျက်သွားရင် ELB က Server A နဲ့ C ဆီပဲ traffic ပို့မယ်။

                3 Servers→2 Healthy Servers

                ဒါကြောင့် service မပြတ်ဘူး။

                နောက်တစ်ခုက multiple Availability Zone တွေသုံးတာပါ။
                ဥပမာ application ကို AZ-A နဲ့ AZ-B နှစ်ခုလုံးမှာ run ထားတယ်ဆိုရင် -
                  AZ-A down သွားရင်
                  AZ-B က service ဆက် run နိုင်တယ်

                ဒါကြောင့် High Availability ရဖို့အတွက် -
                  Multiple servers
                  Load balancer
                  Multiple Availability Zones
                  Auto Scaling

            Fault tolerance:
              Fault domain?
                a section of network that is vulnerable to damage if a critical device or system fails. 
                the purpose of fault domain is that if a failure occurs it will not cascade outside that domain, limiting the damage possible.
                MM Note:
                  Fault Domain ဆိုတာ system တစ်ခုထဲမှာ failure ဖြစ်နိုင်တဲ့ area သို့မဟုတ် group တစ်ခုကို ဆိုလိုပါတယ်။
                  ဥပမာ -
                    Server rack တစ်ခု
                    Network switch တစ်ခု
                    Power supply တစ်ခု
                    Data center တစ်ခု
                    Availability Zone တစ်ခု
                  ဒီ area ထဲက critical component တစ်ခု fail ဖြစ်သွားရင် အဲ့ဒီ area ထဲမှာရှိတဲ့ systems တွေကိုပဲထိခိုက်မယ်။
                  Fault domain ရဲ့ အဓိကရည်ရွယ်ချက်က failure တစ်ခုဖြစ်လာရင် system တစ်ခုလုံးကို မထိခိုက်စေဘဲ damage ကို domain တစ်ခုထဲမှာပဲ ကန့်သတ်ထားဖို့ပါ။
                  ဥပမာ -
                  Server 10 လုံးကို rack တစ်ခုတည်းမှာထားရင် rack power fail ဖြစ်ရင် server 10 လုံးလုံး down သွားနိုင်တယ်
                  Server 10 လုံးကို rack 2 ခုခွဲထားရင် rack တစ်ခု fail ဖြစ်ရင် အခြား rack ကဆက် run နိုင်တယ်

                  AWS မှာတော့ fault domain ကိုခွဲဖို့ multiple Availability Zone တွေသုံးကြတယ်။
                  ဥပမာ app server ကို AZ-A မှာထားပြီး database replica ကို AZ-B မှာထားရင် AZ-A failure ဖြစ်ရင်တောင် system တစ်ခုလုံးမပျက်နိုင်ဘူး။
                  Fault domain နည်းနည်းလေးတွေခွဲထားလေလေ fault tolerance ပိုကောင်းလေလေ ဖြစ်
              Fault level:
                a collection of fault domains.
                scope could be
                  specific servers in a rack
                  an entire rack in a datacenter
                  an entire room ia a datacenter
                  the entire data center building
                its up to the CSPs to define the boundaries of a domain.
                MM Note:
                  Fault Level ဆိုတာ fault domain အများကြီးပေါင်းထားတဲ့ larger failure scope ကိုဆိုလိုပါတယ်။

                  Fault Domain က failure ဖြစ်နိုင်တဲ့ area တစ်ခုဆိုရင် Fault Level က failure ဘယ်လောက်ကြီးမားတဲ့အထိထိခိုက်နိုင်လဲဆိုတဲ့ level ဖြစ်ပါတယ်။

                  ဥပမာ failure scope က -

                  Rack တစ်ခုထဲက specific servers
                  Data center ထဲက rack တစ်ခုလုံး
                  Data center ထဲက room တစ်ခန်းလုံး
                  Data center building တစ်ခုလုံး
                  Availability Zone တစ်ခုလုံး
                  Region တစ်ခုလုံး

                  ဥပမာ power failure တစ်ခုက rack တစ်ခုကိုပဲထိခိုက်ရင် fault level နိမ့်တယ်။

                  ဒါပေမယ့် cooling system ပျက်ပြီး data center တစ်ခုလုံး down သွားရင် fault level ပိုမြင့်တယ်။

                  Amazon Web Services လို CSP တွေက fault domain နဲ့ fault level boundary တွေကို define လုပ်ထားတယ်။

                  ဥပမာ -

                  Different racks
                  Different rooms
                  Different Availability Zones
                  Different Regions

                  ဆိုပြီး infrastructure ကိုခွဲထားတယ်။

                  ဒီလိုခွဲထားတာကြောင့် fault တစ်ခုဖြစ်လာရင် system တစ်ခုလုံးကို cascade မဖြစ်စေဘဲ impact ကိုကန့်သတ်နိုင်တယ်။
                  Region တစ်ခုမှာ major failure ဖြစ်ရင် အခြား Region တွေကို မထိခိုက်စေဘူး။
                  AZ တစ်ခု down သွားရင် အခြား AZ တွေကို မထိခိုက်စေဖို့ design လုပ်ထားတယ်။
              Failover
                Failover ဆိုတာ primary system fail ဖြစ်သွားရင် traffic ကို secondary system ဆီပြောင်းပေးတာပါ
                Primary database ပျက်သွားရင် secondary database က primary အဖြစ်ပြောင်းပြီး service ဆက် run နိုင်တယ်။
                ဒါကြောင့် user တွေအတွက် downtime နည်းသွားတယ်။
                Fault Tolerance ရဖို့အတွက် အများအားဖြင့် -
                  Redundant servers
                    Redundant servers ဆိုတာ server တစ်လုံးပျက်သွားရင် အစားထိုး run ပေးဖို့ server အပိုတွေထားထားတာပါ။
                      ဥပမာ -
                        Server A
                        Server B
                      Server A က main server ဖြစ်ပြီး Server B က backup / standby server ဖြစ်နိုင်တယ်။
                      Server A ပျက်သွားရင် Server B က ဆက် run ပေးမယ်။
                  Replicated databases
                    Replicated databases ဆိုတာ database တစ်ခုရဲ့ data ကို အခြား database တစ်ခုဆီ copy / sync လုပ်ထားတာပါ။
                    ဥပမာ -
                      Primary Database
                      Secondary Database
                    Primary database မှာ data update ဖြစ်တိုင်း secondary database ကို sync လုပ်ပေးတယ်။
                    Primary database ပျက်သွားရင် secondary database ကိုသုံးပြီး service ဆက် run နိုင်တယ်။
                  Multiple Availability Zones
                  Load balancers
                  Automatic failover
                တွေကိုအသုံးပြုကြတယ်။

                Backup restore ဘယ်လောက်မြန်လဲ
                Backup က တကယ် restore လုပ်လို့ရလား
                Live data corrupt မဖြစ်အောင်ဘယ်လိုကာကွယ်ထားလဲ

              AWS spare capacity ကိုသုံးတာဖြစ်ပြီး On-Demand ထက် 90% လောက်သက်သာနိုင်တယ်။
              ဒါပေမယ့် AWS က capacity ပြန်လိုလာရင် instance ကို interrupt လုပ်နိုင်တယ်။
              Flexible workload တွေအတွက်သင့်တော်တယ်။
              ဥပမာ -
                Batch processing
                Video rendering
                Testing
            Dedicated Hosts
              Physical server တစ်လုံးကို ကိုယ်တစ်ယောက်တည်းသီးသန့်သုံးတာပါ။
              အရမ်းစျေးကြီးပေမယ့် existing software license တွေသုံးနိုင်တယ်။
            အကျဉ်းချုပ်ဆိုရင် -
              Short-term = On-Demand
              Long-term predictable = Savings Plans / Reserved Instances
              Cheapest but interruptible = Spot Instances
              Physical server dedicated = Dedicated Hosts
          Scaling:
            Scaling ဆိုတာ workload နဲ့ traffic ပေါ်မူတည်ပြီး resource ကိုတိုးခြင်း၊ လျှော့ခြင်းကို ဆိုလိုပါတယ်။
              key note:
                Traffic ဆိုတာ system ထဲကိုဝင်လာတဲ့ requests, users, data amount ကိုဆိုလို,
                Workload ဆိုတာ system က process လုပ်ရတဲ့ actual work ကိုဆိုလို
                  ဥပမာ user 1,000 ယောက် website ကြည့်တာက traffic များတယ်။
                  ဒါပေမယ့် static page ပဲဆိုရင် workload မများနိုင်ဘူး။
                  တစ်ဖက်မှာ user 10 ယောက်ပဲရှိပေမယ့် video rendering လုပ်နေရင် traffic နည်းပေမယ့် workload အရမ်းများနိုင်တယ်။
            ဒါကြောင့် application က performance ကောင်းနေပြီး unnecessary cost မဖြစ်ဘူး။
            Vertical Scaling
              Server တစ်လုံးရဲ့ CPU, RAM, Storage ကိုတိုးတာပါ။
              ဥပမာ -
                2 vCPU ကနေ 4 vCPU ပြောင်းတာ
                8 GB RAM ကနေ 16 GB RAM ပြောင်းတာ

              Small Instance→Larger Instance

              Vertical scaling ကို scale up / scale down လို့လည်းခေါ်တယ်။
              Database server တွေမှာ vertical scaling ကိုအများကြီးသုံးကြတယ်။
            Horizontal Scaling
              Server အသစ်တွေထပ်ထည့်တာပါ။
              ဥပမာ -
                Web server 1 လုံးကနေ 5 လုံးဖြစ်သွားတာ
                Traffic များလာရင် EC2 instance အသစ်တွေ auto create လုပ်တာ

              1 Server→5 Servers

              Horizontal scaling ကို scale out / scale in လို့လည်းခေါ်တယ်။
                scaling out: add more servers.
                scaling in: remove underutilized server.
              Web applications တွေမှာ horizontal scaling ကိုပိုသုံးကြတယ်။
              AWS မှာတော့ horizontal scaling အတွက် Amazon EC2 Auto Scaling နဲ့ Elastic Load Balancing ကိုသုံးကြတယ်။

              Auto Scaling က traffic ပေါ်မူတည်ပြီး instance အသစ်တွေ create / remove လုပ်ပေးတယ်
              Load Balancer က user requests တွေကို server အများကြီးဆီ distribute လုပ်ပေးတယ်
            Note: Server တွေက request အရမ်းများလာရင် timeout, slow response, outage ဖြစ်နိုင်တယ်။ Auto Scaling က အဲ့ဒီ problem ကိုကာကွယ်ပေးတယ်
            Dynamic Scaling
              Dynamic scaling က real-time traffic change ကိုကြည့်ပြီး instance တိုး / လျှော့ လုပ်တာပါ။
            Predictive Scaling
              Predictive scaling က historical traffic pattern ကိုကြည့်ပြီး ကြိုတင် scale လုပ်တာပါ။
              ဥပမာ -
                Website က နေ့တိုင်း မနက် 9 နာရီမှာ traffic တက်တယ်
                Auto Scaling က 8:50 မှာတင် instance အသစ်တွေကြိုဖွင့်ထားမယ်
            Auto Scaling Group (ASG):
              ASG မှာ capacity limit 3 မျိုးသတ်မှတ်တယ် -
                Minimum = အနည်းဆုံး instance ဘယ်နှစ်လုံး run နေရမလဲ
                Desired = Normal traffic အတွက် ideal instance count
                Maximum = ဘယ်နှစ်လုံးထက်မပိုရဘူးဆိုတဲ့ limit
                ဥပမာ -
                  Minimum = 2
                  Desired = 2
                  Maximum = 10
                  2≤Instances≤10

                  ဒီ setup မှာ normal အချိန် 2 instances run မယ်။
                  Traffic တက်လာရင် Auto Scaling က 3, 4, 5 instance ဖြစ်အောင်တိုးမယ်။
                  အများဆုံး 10 instances ထိပဲတိုးနိုင်မယ်။
            Summary:
              Vertical Scaling = Server တစ်လုံးကို ပိုကြီးအောင်လုပ်တာ
              Horizontal Scaling = Server အရေအတွက်တိုးတာ
              Dynamic Scaling = Traffic တက်မှ auto တိုးတာ
              Predictive Scaling = Traffic တက်မယ့်အချိန်ကို ကြိုတွက်ပြီး တိုးတာ
        ELB:
          incoming traffic တွေကို EC2 instances အများကြီးကြားမှာ distribute လုပ်ပေးတဲ့ service
            Routing Methods:
              Round Robin
                Request တွေကို server တစ်လုံးပြီးတစ်လုံး အလှည့်ကျ distribute လုပ်တယ်။
                ဥပမာ -
                Request 1 → Server A
                Request 2 → Server B
                Request 3 → Server C
                Request 4 → Server A
              Least Connections
                Active connection အနည်းဆုံးရှိတဲ့ server ကို request ပို့တယ်။
                ဥပမာ server A မှာ 50 users ရှိပြီး server B မှာ 10 users ပဲရှိရင် request အသစ်ကို server B ဆီပို့မယ်။
              IP Hash
                Client IP ကိုကြည့်ပြီး user တစ်ယောက်ကို server တစ်လုံးတည်းဆီ consistent ပို့တယ်။
                ဥပမာ user တစ်ယောက်က session data တူညီနေဖို့ server တစ်လုံးတည်းကိုပဲရောက်နေစေချင်တဲ့အခါ သုံးတယ်။
              Least Response Time
                Response အမြန်ဆုံးပေးနိုင်တဲ့ server ကို request ပို့တယ်။
                ဥပမာ server A က 100ms နဲ့ပြန်တယ်၊ server B က 30ms နဲ့ပြန်တယ်ဆိုရင် request အသစ်ကို server B ဆီပို့မယ်။

              ans: Incoming web traffic

          Application architecture:
            two main approaches to connnecting components.
            Monolithic:
              tightly coupled, as a single unit.
              if one fails, other can go down.
            Microservices:
              loosely coupled, communicate through messaging.
              if one fails, other continue work.

            Amazon EventBridge:
              Serverless event routing between apps, AWS services, and third-party software.
            Amazon SNS:

            Amazon SQS:
              Message queuing for reliable component communication.

        AWS Serverless:
          You focus on code while AWS handles servers, scaling, and maintenance.
          server infrastructure ကိုကိုယ်တိုင် manage မလုပ်ဘဲ application code ပဲရေးပြီး run လုပ်တာပါ။

              EC2: You manage the server, install software, and handle updates
              Serverless: You only write code, while AWS handles everything else.
        AWS Lambda:
          AWS Lambda is a serverless compute service that runs code in response to events. No servers to manage, automatic scaling, 
          and you pay only for compute time used (down to the millisecond).
          AWS Lambda က event ဖြစ်လာတဲ့အခါ code ကို automatically run ပေးတဲ့ serverless compute service ဖြစ်ပါတယ်။
            Server create, OS install, patching, scaling တွေကို AWS က manage လုပ်ပေးတယ်။
            Developer က code ပဲရေးရတယ်။
            Lambda ရဲ့ အဓိက advantage က -
              No servers to manage
              Automatic scaling
              Pay only when code runs
              Millisecond-level billing
              How Lambda Works
            1. Upload Code
            Code ကို Lambda function အဖြစ် upload လုပ်တယ်။
            2. Set Triggers
            ဘယ် event ဖြစ်လာရင် function run မလဲဆိုတာသတ်မှတ်တယ်။
            ဥပမာ -
              File upload to Amazon S3
              HTTP request from API
              Schedule time
              Database update
              Message queue event
            3. Code Runs Automatically
            Event ဖြစ်တာနဲ့ Lambda က function ကို run လုပ်ပေးတယ်။
            Traffic များလာရင် Lambda က auto scale လုပ်တယ်။
            4. Pay Only for Usage
            Code run နေတဲ့အချိန်နဲ့ memory usage အတွက်ပဲ billing ဖြစ်တယ်။
            Code မ run နေရင် almost no cost ဖြစ်တယ်။

        Networking:
          Networking refers to interconnected devices that exchange data and resources.
          The main building blocks are:
            Amazon VPC (Virtual Private Cloud): 
              A logically isolated section of AWS where you launch resources in a virtual network you define.
              AWS ထဲမှာ ကိုယ်ပိုင် private network တစ်ခုတည်ဆောက်တာပါ။
            Subnets:
              Sections of a VPC used to organize resources. Can be public (internet-facing) or private (internal only).
              VPC ထဲက smaller network section ဖြစ်ပါတယ်။
              Resources တွေကို organize လုပ်ဖို့ subnet တွေခွဲတယ်။
              Subnet အဓိက ၂ မျိုးရှိတယ် -
                Public Subnet
                  Internet ကနေတိုက်ရိုက် access လုပ်နိုင်တဲ့ subnet ဖြစ်တယ်။
                Private Subnet
                  Internet ကနေတိုက်ရိုက် access မရတဲ့ subnet ဖြစ်တယ်။
                  ဥပမာ -
                    Database server
                    Internal application server
                    Backend services
            Connectivity:
              AWS resources တွေနဲ့ connect လုပ်ဖို့ နည်းလမ်းအမျိုးမျိုးပေးထားတယ်။
              Public internet ကနေ connect လုပ်တာရှိသလို private dedicated connection တွေလည်းရှိတယ်။
              Internet Gateway:
                Internet Gateway က VPC နဲ့ public internet ကြား connection ဖြစ်ပါတယ်။
                ဥပမာ public web server တစ်ခု internet ကနေ access ရဖို့ Internet Gateway လိုတယ်။
              Virtual Private Gateway:
                Virtual Private Gateway က company private network နဲ့ VPC ကြား encrypted VPN connection တည်ဆောက်ပေးတယ်။
                ဒီ connection က approved network တွေကပဲ access ရမယ်။
                ဒါကြောင့် secure private connection ရတယ်။
                ဥပမာ office network ကနေ AWS VPC ထဲက database ကို VPN နဲ့ connect လုပ်နိုင်တယ်။
              AWS Direct Connect:
                AWS Direct Connect က company datacenter နဲ့ AWS ကြား dedicated/physical private line တည်ဆောက်ပေးတယ်။
                Public internet မသုံးဘဲ direct private connection ဖြစ်တယ်။
            Cloud Subnet and Access:
              Network Traffic:
                data travels across networks as packets.
                when a packet enters a vpc through an Internet Gateway.
              Network Access Control Lists(ACLs):
                ACLs are firewalls that control inbound and outbound traffic at the subnet level.
                Packets on the approved list pass through; others are denied. 
                ACLs use stateless packet filtering: no memory of previous requests.
              Security Groups:
                Security Groups are firewalls that control traffic for specific EC2 instances. 
                By default, they deny all inbound traffic and allow all outbound. 
                Security Groups use stateful packet filtering.
                Which statement about Security Groups is correct?|
                  A. They operate at the subnet level
                  B. They are stateless
                  C. They control traffic for specific EC2 instances
                  D. They deny all outbound traffic by default
                  Correct ans: C.
            Global Networking:
              user တွေဘယ်နိုင်ငံကဝင်ဝင် application ကိုမြန်မြန်နဲ့ reliable ဖြစ်အောင် route လုပ်ပေးတဲ့ networking services
              DNS:
                DNS (Domain Name System) က internet phone book လိုမျိုး
                User က website name ရိုက်ထည့်လိုက်ရင် DNS က domain name ကို IP address အဖြစ်ပြောင်းပေးတယ်။
                Computer ကတော့ actual IP address နဲ့ connect လုပ်တယ်။
              Amazon Route 53
                Amazon Route 53 က AWS ရဲ့ DNS service ဖြစ်တယ်။
                Route 53 က user request ကို correct destination ဆီ route လုပ်ပေးတယ်။
                Flow က ဒီလိုဖြစ်တယ် -
                  User က website domain ကို request လုပ်တယ်
                  Amazon Route 53 က domain name ကို IP address resolve လုပ်ပေးတယ်
                  Request က user နဲ့အနီးဆုံး Amazon CloudFront Edge Location ဆီသွားတယ်
                  CloudFront က backend application ဆီ request ပို့တယ်
                  Request က Load Balancer ဆီရောက်တယ်
                  Load Balancer က healthy EC2 instance ဆီ request ပို့တယ်
                  Response ကို user ဆီပြန်ပို့တယ်

                  User→Route 53→CloudFront→Load Balancer→EC2
                  CloudFront က frequently requested content တွေကို Edge Location မှာ cache ထားပေးတယ်။
                  ဒါကြောင့် user နဲ့ data source ကြား distance နည်းပြီး website load ပိုမြန်တယ်။
              AWS Global Accelerator
                AWS Global Accelerator က traffic ကို public internet မသုံးဘဲ AWS private global network ပေါ်ကနေ route လုပ်ပေးတယ်။
              Global Architectures
                Direct Connect Failover:
                  ဒီ architecture pattern က high-bandwidth private connection လိုတဲ့ company တွေအတွက်အသုံးများတယ်
                  Single connection တစ်ခုတည်းသုံးရင် အဲ့ဒီ line ပျက်သွားတဲ့အခါ service down နိုင်တယ်။
                  ဒါကြောင့် multiple Direct Connect connections သုံးပြီး failover setup လုပ်တယ်။
                Multi-Region Content Delivery:
                  pattern က global users တွေအတွက် low latency နဲ့ content ပို့ချင်တဲ့အခါသုံးတယ်။
                  User က website ကို request လုပ်တဲ့အခါ -

                  User request က Amazon Route 53 ဆီသွားတယ်
                  Route 53 က nearest region သို့မဟုတ် best endpoint ကိုရွေးတယ်
                  Request က nearest Amazon CloudFront Edge Location ဆီသွားတယ်
                  Cached content ရှိရင် CloudFront က direct response ပေးတယ်
                  Cache မရှိရင် origin server ဆီသွားပြီး content ယူတယ်
                  Origin server ကို multiple Availability Zones မှာထားပြီး high availability ရအောင်လုပ်တယ်
        AWS Storage:
          provides multiple storage options to meet different needs: 
          block storage for EC2 instances, 
          object storage for files and media, and 
          file storage for shared access across applications.
          Storage Types:
            Block Storage:
              Low-latency storage volumes that attach to EC2 instances like physical hard drives.
                EC2 Instance Store: Temporary, high-performance storage
                  EC2 instance stop / terminate ဖြစ်သွားရင် data ပျောက်နိုင်တယ်။
                Amazon EBS: Persistent block storage volumes
                  EC2 restart လုပ်လည်း data မပျောက်ဘူး။
            Object Storage:
              Manages data as objects with unlimited scalability for unstructured data like documents, images, and videos.
                Amazon S3: Store and retrieve any amount of data
                  unstructured data တွေအတွက်သင့်တော်
                  Which AWS service can be used as the origin for cached content in CloudFront?
                    A. Amazon S3
                    B. Amazon Route 53
                    C. AWS Direct Connect
                    D. Amazon VPC
                    Correct Ans: A
                    Reason: Amazon CloudFront needs an origin server or storage location to fetch content from, and 
                    Amazon S3 is a common origin for static files like images, videos, and website assets.
            File Storage:
              Shared file systems accessible over networks by multiple users and applications simultaneously.
                Amazon EFS: Scalable NFS file system, Linux servers တွေအတွက်
                Amazon FSx: Managed file storage, Windows, Lustre, NetApp, OpenZFS စတဲ့ specialized file systems တွေအတွက်
          Shared Responsibility for Storage:
            Storage service အလိုက် AWS နဲ့ customer responsibility ကမတူဘူး။
              Fully managed: AWS handles everything. You only manage data, access, and configuration, eg: s3
              Managed: AWS handles infrastructure. You only manage backups and encryption, eg: amazon rds
              Unmanaged: You handle everything. You manage data management, backups, and encryption, eg: ec2 with attached storage

        Business Continuity Plan (BCP):
          a document that outlines how a business will continue operation durying an unplanned disruption in services.
          MM Note:
            BCP ဆိုတာ business တစ်ခုမှာ unexpected problem (server down, data center ပျက်, network outage) ဖြစ်လာတဲ့အချိန်မှာ လုပ်ငန်းကို ဆက်လက် run နိုင်အောင် 
            ဘယ်လိုလုပ်မလဲဆိုတာရေးထားတဲ့ plan/document ဖြစ်ပါတယ်။
          Recovery Point Objectgive(RPO):
            the max acceptable amt of data loss after an unplanned data-loss incident, expressed as an amt of time.
            How much data are you willing to lose?
            RPO က data loss ကိုဘယ်လောက်ထိ ခွင့်ပြုမလဲ ဆိုတာကို ပြောတာပါ။
            Time နဲ့တိုင်းတာတယ်။
            ဥပမာ -
              RPO = 5 minutes
              → အများဆုံး 5 minutes data ပဲပျောက်လို့ရတယ်
              → Backup / replication ကို 5 mins တစ်ခါလုပ်ရမယ်
              RPO = 1 hour
              → 1 hour data loss လက်ခံနိုင်တယ်
              🔑 RPO နည်းလေလေ → data protection ပိုကောင်းလေလေ → cost ပိုများနိုင်တယ်
          Recovery Time Objective(RTO):
            the max amt of downtime your business can tolerate without incurring a significant financial loss.
            How much time are you willing to go down?
            RTO က system down ဖြစ်တဲ့အချိန် ဘယ်လောက်ကြာမှ ပြန်လာနိုင်မလဲ ဆိုတာပါ။
            👉 “Service ဘယ်လောက်ကြာ down ဖြစ်လို့ရလဲ?”

            ဥပမာ -

            RTO = 10 minutes
            → 10 minutes အတွင်း system ပြန်တက်ရမယ်
            RTO = 4 hours
            → 4 hours downtime လက်ခံနိုင်တယ်

            🔑 RTO နည်းလေလေ → recovery speed မြန်လေလေ → infrastructure cost ပိုများနိုင်တယ်
            Summary:
              RPO = Data loss tolerance
              RTO = Downtime tolerance

              RPO=Data Loss Time,RTO=Recovery Time

              Simple Example
              Company တစ်ခုမှာ -

              RPO = 5 minutes
              RTO = 10 minutes

              ဆိုရင် -
                Data loss = အများဆုံး 5 mins ပဲ
                System down = အများဆုံး 10 mins ပဲ
              ဆိုပြီး design လုပ်ရမယ်။
        Disaster Recovery Options:
          muliple options for recovery
            Backup & Restore > Pilot Light > Warm Standy > Multi-site Active/Active
            Backup & Restore:
              You backup your data and restore it to new infrastructure
              RPO/RTO: Hours
              . Lower priority use cases
              . Restore data after event
              . Deploy resources after event . Start & and scale resources after event
              . Cost $
            Pilot Light:
              RPO/RTO: 10mins
              Data is replicated to another region with the minimal services running
              . Less stringent RTO &RPO
              . Core Services
              . Cost $$
            Warm Standy:
              RPO/RTO: Minutes
              Scaled down copy of your infrastructure running ready to scale up
              . Business Critical Services
              . Scale resources after event
              . Cost $$$
            Multi-site Active/Active:
              RPO/RTO: Real-time
              Scaled up copy of your infrastructure in another region
              · Zero downtime
              · Nero zero loss
              . Mission Critical Services
              . Cost $$$$
